# 什么是 Context Switching

Context Switching（上下文切换）指的是操作系统在暂停当前执行实体并恢复另一个执行实体时所做的一系列状态保存与恢复动作。

> [!NOTE]
> 这里的“执行实体”通常可以是：
> - 进程（process）
> - 线程（thread）
> - 内核线程 / 用户线程
> - 中断处理上下文

---

# 为什么需要上下文切换

CPU 同一时刻在一个核心上只能真正执行一个线程。但是在用户层面，系统不能一直执行某一个线程，然后其他线程未响应。因此操作系统需要做到：

- **并发**：让多个任务“看起来”同时运行
- **公平**：不能让一个任务独占 CPU
- **响应性**：交互式任务要尽快得到执行
- **隔离性**：不同进程彼此独立

因此 OS 会不断在不同任务之间切换。

---

# “上下文”到底包含什么

切换时需要保存和恢复的内容通常包括：
* [CPU 寄存器](./what-is-register.md)
* 栈相关信息 &rarr; 线程运行依赖自己的栈，切换线程时，必须切换到对应线程自己的栈
* 地址空间信息 &rarr; 如果进程切换那么涉及虚拟地址空间切换
* 调度器元信息 &rarr; 比如当前状态（running / ready / blocked）、优先级、时间片等

---

# 进程切换 vs 线程切换

## 线程切换

线程切换通常需要管理：

- 寄存器
- 栈
- 调度状态

如果两个线程属于同一进程，它们共享：

- 地址空间
- 页表
- 打开的文件描述符
- 大部分进程资源

所以线程切换通常比进程切换便宜。

## 进程切换

进程切换除了线程管理的内容，还可能需要：

- 切换页表
- 更换地址空间
- 影响 TLB
- 更重的 cache / locality 损失

---

# 上下文切换发生时机

## 时间片用完

抢占式调度器会给线程一个时间片。时间片耗尽后，调度器可能把 CPU 分给别的线程。

## 线程主动阻塞

例如线程调用阻塞型指令，线程会进入阻塞态，CPU 会切给别的可运行线程：

- `sleep`
- `read` 等待 I/O
- `futex`
- `pthread_mutex_lock` 被阻塞
- `wait`
- `epoll_wait`

## 更高优先级任务到来

如果来了更高优先级线程，当前线程可能被抢占。

## 中断 / 异常

- 定时器中断
- I/O 中断
- page fault
- syscall 陷入内核后可能发生调度

---

# Context Switch 的成本

* 寄存器保存和恢复
* 内核调度路径开销 &rarr; 进入内核、调度决策、队列操作、状态更新，本身就有成本
* Cache miss &rarr; 新线程用新数据，原本的cache容易miss
* 进程切换导致地址空间变化
* CPU pipeline / branch predictor 影响
* 内存局部性被破坏

---

# Voluntary vs Involuntary Context Switch

Voluntary Context Switch 线程是“自愿”让出 CPU，常见于：
- 等待锁
- 等待 I/O
- `sleep`
- 阻塞系统调用

Involuntary Context Switch 线程并不想让出 CPU，但被调度器抢占，例如：

- 时间片到
- 更高优先级任务抢占

---

# CPU-bound & I/O-bound

## CPU-bound

任务主要耗费 CPU 计算资源，例如：

- 压缩
- 加密
- 图像处理
- 数值计算

这类任务更容易吃满时间片，被抢占切换。

## I/O-bound

任务主要在等外设或网络，例如：

- 磁盘读写
- 网络请求
- 数据库连接等待

这类任务经常主动阻塞，因此也会频繁发生切换。

---

# 如何减少不必要的 Context Switch

## 减少锁竞争

- 缩短临界区
- 减少共享状态
- 使用更好的并发结构
- 分片 / sharding

## 控制线程数

- 不要盲目一个请求一个线程
- 使用线程池
- 线程数与 CPU / workload 匹配

## 减少阻塞 I/O

- 非阻塞 I/O
- `epoll`
- async 模型
- 批处理

## 提高局部性

- CPU affinity
- 避免频繁跨核迁移
- NUMA-aware 分配

## 选择合适同步原语

- 自旋锁
- mutex
- rwlock
- lock-free / wait-free 结构
