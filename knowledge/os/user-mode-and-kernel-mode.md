# 什么是用户态和内核态

操作系统通常会把 CPU 的执行权限分成不同级别。  
- **用户态（User Mode）** &rarr; 权限较低，普通应用程序运行的状态
- **内核态（Kernel Mode）** &rarr; 权限较高，操作系统内核运行的状态

---

# 为什么要区分用户态和内核态

之所以要这样设计，是为了让系统更安全、更稳定。如果没有权限隔离，普通程序就可以随意做下面这些事，对系统造成严重破坏：

- 直接读写任意物理内存
- 直接访问磁盘控制器
- 随意操作网卡
- 关闭中断
- 修改页表
- 破坏其他进程的数据

---

# 特点

| 模式 | 特点 |
| --- | --- |
| 用户态 | 权限受限<br/>不能直接访问特权指令<br/>不能直接操作硬件<br/>不能任意访问内核内存<br/>需要通过系统调用向内核请求服务 |
| 内核态 | 具有最高权限<br/>可以执行特权指令<br/>可以访问硬件资源<br/>可以管理进程、线程、内存等系统资源 |

---

# 用户态和内核态分别在运行什么

一般而言写的 C++ / Python 代码，通常默认都是在用户态运行。
- 浏览器
- 应用软件
- 数据库的用户逻辑
- 命令行程序

内核态通常运行的是操作系统内核代码。
- 进程调度
- 内存管理
- 文件系统
- 网络协议栈
- 中断处理
- 设备驱动

---

# 特权指令 (Privileged Instruction)

有一些 CPU 指令不能让普通应用随便执行，这些就叫 **特权指令**。一般只能在内核态执行。

- 修改页表或地址转换相关寄存器
- 开关中断
- 执行 I/O 端口操作
- 改变 CPU 某些控制寄存器
- 触发某些底层硬件控制行为

如果用户态程序试图直接执行这些指令，通常会触发异常，由操作系统接管。

---

# 系统调用 (System Call)

应用代码在用户态执行，但遇到需要操作系统帮忙的事情时，就通过 syscall 进入内核态。

- 读/写文件：`read`/`write`
- 创建进程：`fork`
- 退出进程：`exit`
- 分配内存：`mmap`
- 建立网络连接：`connect`

---

# 什么情况下会发生转换

## syscall

程序正常运行需要调用操作系统级别的操作时会使用system call。

## 异常（exception）

程序运行过程中如果出现异常情况，也会切入内核态。

- 缺页异常（page fault）
- 除零错误
- 非法指令
- 访问非法地址
- 用户态执行特权指令

## 中断（interrupt）

外部硬件事件也会让 CPU 切入内核态。中断发生时，CPU 会暂停当前执行流，转去执行内核中的中断处理逻辑。

- 定时器中断
- 键盘输入
- 网卡收到数据包
- 磁盘 I/O 完成
- 其他设备中断

---

# 用户态和内核态切换时，CPU 做了什么

- 切换特权级
    - 从 user mode 切到 kernel mode
- 保存当前现场
    - 保存当前程序计数器、部分寄存器、状态寄存器等，这样之后才能继续回到原来的执行点
- 跳到内核指定入口
    - 如果是系统调用、异常、硬件中断，CPU 会根据中断向量表/陷阱表跳到对应的内核处理函数
- 切换到内核栈
    - 用户程序有自己的用户栈
    - 进入内核后，CPU/系统会使用当前线程对应的内核栈来处理系统调用或中断
- 执行内核代码
- 恢复现场并返回
    - 内核处理完后，恢复之前保存的寄存器和执行位置
    - 执行返回指令，CPU 再回到 user mode

进入内核态后，通常不再使用原来的用户栈，而是切到对应线程的内核栈。因此在进入内核态之前需要**保存一些和context相关的寄存器状态到内存**。

## 保存的内容和位置

由于是调度线程，因此一般是保存到TCB中。详情参考[PCB & TCB](./processes-and-threads.md#control-block)。

---

# 用户栈和内核栈

每个线程通常至少会涉及两套栈：
- **用户栈**：线程在用户态运行时使用
- **内核栈**：线程陷入内核态后使用

内核不能完全信任用户空间的数据和栈内容。如果继续用用户栈，可能会有安全性和稳定性问题。所以当发生 syscall / interrupt / exception 时，CPU 和 OS 会切到内核栈执行内核逻辑。

---

# 用户态/内核态切换 ≠ context switch

用户态/内核态切换强调的是权限级别发生变化，实际上可能还是同一个线程继续执行。

反之上下文切换是更换线程，也不一定涉及到system call / exception / interrupt等状态切换。

---

# 系统调用一定会导致线程切换吗

不一定。例如`getpid()`等一些非阻塞的操作，它们通常只是：用户态进入内核态 &rarr; 内核做一点工作 &rarr; 立刻返回用户态，整个过程中线程可能没有被切走。

但也有些系统调用会导致阻塞，比如：

- `read` 等数据
- `accept` 等连接
- `sleep`
- `futex` 等待锁

这时线程可能会被挂起，然后调度器切给别的线程。

---

# 为什么减少 syscall 很重要

因为每次 syscall 都会跨越用户态和内核态边界。所以常见优化思路包括：

- 批量读写，减少频繁小 I/O
- 使用缓冲
- 使用 `epoll` 替代低效轮询
- 使用 `mmap` 减少显式 read/write
- 采用零拷贝技术
- 使用更高效的异步 I/O 机制
